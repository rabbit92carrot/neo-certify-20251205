## 1. 제품 개요

### 1.1 제품 정의
PDO threads(미용성형용 의료기기)의 생산부터 최종 소비까지 전 과정의 이력을 추적하여 정품을 인증하는 시스템

### 1.2 핵심 특징
- **환자 개입 최소화**: 시술 후 카카오 알림톡으로 인증 결과만 수신
- **조작 단순화**: 제품 종류 선택 + 수량 입력만으로 모든 거래 처리
- **가상 식별코드**: 물리적 개별 추적이 어려운 제품 특성상 DB 기반 가상 코드로 통계적 관리
- **즉시 이관**: Pending 없이 모든 이관 즉시 처리, 오류 시 24시간 내 회수

### 1.3 서비스 형태
- **1차**: 자사(네오닥터) 전용 SaaS
- **향후**: 타 제조사 확장 가능한 플랫폼으로 발전

### 1.4 장기 비전
별도 법인으로 분리하여 여러 제조사가 이용 가능한 지속가능한 SaaS 비즈니스 모델로 확장

## 2. 배경 및 문제 정의

### 2.1 배경
- 자사 PDO threads 제품과 유사한 형태의 제품이 시장에 존재
- 특허로 보호 중이나 형태 모방 제품으로 인한 잠재적 리스크 존재
- 타 분야(고주파 리프팅 등)에서 유사품 문제로 정품인증 도입 사례 다수
- 소포장(1파우치 2개입) 특성상 실물 인증/QR 방식 부적합

### 2.2 핵심 문제 (동등 중요도)

| 문제 | 설명 |
|------|------|
| **환자의 정보 비대칭** | 시술 전후로 제품을 직접 볼 수 없고, 정품 여부 확인 방법 전무 |
| **브랜드 가치 보호** | R&D 투자 및 특허 확보 제품의 시장 파이를 유사품에 빼앗길 리스크 |
| **유통 이력 투명성** | 기존 월말 결산 방식의 추적 시스템으로 인한 누락 가능성 |

### 2.3 이해관계자별 현황

| 이해관계자 | 현재 문제 |
|------------|-----------|
| **환자** | 정보 비대칭 - 병원의 말을 믿을 수밖에 없는 구조 |
| **병원 (양심적)** | 일부 비양심적 병원으로 인한 시장 교란의 간접 피해 |
| **유통사** | 추가 이력 관리 업무 부담 (수용 여부 미검증) |
| **제조사** | 유사품으로 인한 시장 점유율 하락 리스크 |

## 4. 타겟 사용자

### 4.1 사용자 유형

| 사용자 유형 | 역할 | 예상 조직 수 | 주요 액션 |
|-------------|------|--------------|-----------|
| **제조사** | 생산 및 판매 | 1개 | 제품 등록, 생산, 출고, 회수 |
| **유통사** | 유통 및 판매 | 2~3개 | 출고, 회수 |
| **병원** | 시술 및 인증 | 다수 | 시술 등록, 회수 |
| **환자** | 인증 결과 수신 | - | 시스템 미사용, 알림톡만 수신 |
| **시스템 관리자** | 전체 시스템 관리 | 자사 직원 | 모니터링, 계정 관리, 이력 조회 |

### 4.2 계정 구조

#### 1조직 1계정 체제
- **조직 = 계정**: 조직과 사용자를 구분하지 않음
- **공유 사용**: 1개 계정을 조직 내 여러 직원이 공유
- **로그인 ID**: 대표 이메일 사용
- **이력 기록**: "누가" 작업했는지 기록하지 않음 (향후 필요 시 변경)

#### 가입 프로세스
1. 조직 유형 선택 (제조사/유통사/병원)
2. 대표 이메일, 비밀번호, 조직 정보 입력
   - 조직명, 사업자 등록번호, 사업자 등록증 파일
   - 대표자명, 대표 연락처, 주소
3. 사업자 등록번호로 기존 조직 조회
   - 조직 존재 → 중복 가입 불가 에러
   - 조직 미존재 → 신규 조직 생성
4. 관리자 승인 (1차 개발에서는 임시 비활성화, 로직은 구현)

### 4.3 UX 원칙
- **직관적 UI**: IT 역량과 무관하게 사용 가능
- **박스형 UI 선택**: 제품 선택 → 수량 입력 → 확인
- **최소 입력**: 병원 사용자의 편의성 극대화

## 5. 유통 구조 및 플로우

### 5.1 유통 구조
```
제조사 → 유통사 → 병원 → 환자
         ↓
       유통사 (다단계 가능)
```
- **기본**: 제조사 → 유통사 → 병원 → 환자
- **직접 판매**: 제조사 → 병원 → 환자
- **다단계 유통**: 유통사 → 유통사 가능
- **해외 확장**: 향후 현지 메신저 연동 고려

### 5.2 제품 식별 체계

#### 현재 체계의 한계
- **Lot 번호**: 동일 Lot 내 동일 번호 (개별 추적 불가)
- **UDI-DI**: 제품 종류 + 포장 단위별 동일
- **UDI-PI**: 사용기한 + Lot 번호 결합 (개별 추적 불가)

#### 해결 방안
- Lot 단위 총 수량에 **가상 일련번호** 부여 (DB 관리)
- 완전한 개별 추적은 보장하지 않으나 **통계적 관리 보장**
- **기본 FIFO 처리**: 먼저 생산된 Lot에서 우선 출고
- **Lot 선택 옵션**: 제조사 판매 시에만 특정 Lot 선택 가능 (선택사항)

### 5.3 정상 플로우

| 단계 | 주체 | 액션 | 소유권 |
|------|------|------|--------|
| 생산 | 제조사 | 제품 선택 → 수량 입력 → 등록 | 제조사 |
| 제조사→유통사 | 제조사 | 유통사 선택 → 제품/수량 장바구니 → 출고 (Lot 선택 옵션) | 유통사 (즉시) |
| 유통사→유통사 | 유통사 | 유통사 선택 → 제품/수량 장바구니 → 출고 | 유통사 (즉시) |
| 유통사→병원 | 유통사 | 병원 선택 → 제품/수량 장바구니 → 출고 | 병원 (즉시) |
| 제조사→병원 | 제조사 | 병원 선택 → 제품/수량 장바구니 → 출고 (Lot 선택 옵션) | 병원 (즉시) |
| 시술 | 병원 | 제품/수량 장바구니 → 환자 전화번호 → 확인 | 환자 (즉시) |
| 인증 | 시스템 | 알림톡 자동 발송 | - |

> **즉시 이관 원칙**: Pending 없이 모든 출고는 즉시 소유권 이전됨

### 5.4 예외 플로우

| 시나리오 | 처리 방법 |
|----------|-----------|
| **이관 오류 회수** | 발송자가 24시간 이내 회수 (사유 필수), 환자 회수 시 알림 발송 |
| **폐기** | 별도 메뉴 → 제품/수량 선택 → 사유 입력 (2차) |
| **Lot 수량 오입력** | 코드 폐기 기능으로 대응 (2차) |
| **알림톡 발송 실패** | SMS 대체 발송 (2차) |

> **반품 기능**: 향후 기능으로 보류

## 8. 사용자 스토리 / 유스케이스

### 8.1 제품 종류 등록 (제조사)
```
1. 제조사가 제품 관리 메뉴 접속
2. "제품 등록" 클릭
3. 제품명, UDI-DI, 모델명 입력
4. 등록 완료
```

### 8.2 Lot 생산 등록 (제조사)
```
1. 생산 등록 메뉴 접속
2. 제품 종류 선택 (박스형 UI, 비활성화 제품 미노출)
3. 수량 입력
4. 자동 생성 항목 확인/수정
   - Lot 번호: 환경설정 정규식 기반 (기본: ND + 모델숫자5자리 + yymmdd)
   - 생산일자: 현재 날짜 (수정 가능)
   - 사용기한: 생산일 + 설정기간 (수정 가능)
5. 등록 확인 → DB에 수량만큼 고유 식별번호 자동 생성
```

### 8.3 출고 (제조사/유통사/병원)
```
1. 출고 메뉴 접속
2. 대상 조직(유통사/병원/환자) 선택
3. 제품 종류 선택
4. (제조사만) 특정 Lot 선택 옵션 - 미선택 시 FIFO 자동 처리
5. 수량 입력 → 장바구니에 담기 (같은 제품 중복 시 합산)
6. 장바구니에서 수량 수정 가능
7. 필요한 만큼 3-6 반복
8. 장바구니 확인 → 최종 출고 결정
9. 즉시 소유권 이전 + 이관 뭉치 생성
```

### 8.4 시술 등록 (병원)
```
1. 시술 등록 메뉴 접속
2. 제품 종류 선택 → 수량 입력 → 장바구니에 담기
3. 여러 제품 사용 시 반복
4. 환자 전화번호 입력
5. 확인 → 소유권 환자에게 이전 (전화번호가 소유자 ID)
6. DB에 알림톡 메시지 기록 (1차: Mock 페이지에 표시)
```

### 8.5 이관 오류 회수
```
1. 이관 이력 메뉴 접속
2. 이관 뭉치 목록 확인 (접기/펼치기 UI)
   - 접힌 상태: 발송일, 대상 조직, 제품 종류별 수량
   - 펼친 상태: Lot별 수량, 제조일자, 개별 식별코드
3. 회수할 이관 뭉치 선택 (발송 시점부터 24시간 이내만 가능)
4. 회수 사유 입력 (필수)
5. 확인 → 소유권 발송자에게 복귀
6. 이력에 회수 기록 추가 (별도 색상 표시)
7. 환자 회수 시 → 환자에게 회수 알림 발송 (Mock 페이지에 기록)
※ 24시간 경과 시: "24시간 경과하여 처리할 수 없습니다. 관리자에게 연락해주세요."
```

### 8.6 재고 조회
```
1. 재고 조회 메뉴 접속
2. 필터 설정
   - 제품 종류 선택 (드롭다운)
   - Lot 번호 선택
   - 기간 필터
3. 메인 뷰: 제품 종류별 총 재고량 확인
4. 제품 선택 → 상세 뷰: 일자별(생산일) 수량 확인
```

### 8.7 이관 이력 조회 (뭉치 단위)
```
1. 이관 이력 메뉴 접속
2. 뭉치 단위 목록 표시 (무한 스크롤, 오래된 순)
3. 접힌 상태에서 확인 가능 항목:
   - 발송일시
   - 대상 조직명
   - 제품 종류별 수량
   - 회수 여부 (회수 시 별도 색상 + 사유)
4. 뭉치 클릭 → 펼침
5. 펼친 상태에서 추가 확인 가능:
   - Lot 번호별 수량
   - 제조일자
   - 사용기한
   - 개별 식별코드 목록
```

### 8.8 거래 이력 조회
```
1. 이력 조회 메뉴 접속
2. 필터 설정
   - 기간 필터
   - 거래 유형 (출고/시술 등)
   - 상대 조직
3. 자신의 거래/생산 이력 확인
4. 회수 이력은 별도 색상으로 표시
```

### 8.9 이력 조회 (관리자)
```
1. 이력 조회 메뉴 접속
2. 필터 설정
   - 제조사 선택
   - 제품 종류 다중 선택
   - 기간, 상태, 소유자, 최초 생산자 등
3. 조회 실행
4. Excel형 테이블로 결과 확인 (무한 스크롤)
   - 개별 식별코드, 생산일자, 현재 상태, 현재 소유자
   - 이동 이력 (회수 이력 별도 색상 표시 + 사유)
```

## 11. 에러 메시지 정의

| 상황 | 에러 메시지 |
|------|-------------|

| 잘못된 입력값 | {필드명}을(를) 올바르게 입력해주세요. |

## 15. 비즈니스 로직 상세

### 15.1 FIFO 처리 규칙
- **기본 원칙**: 모든 출고는 먼저 생산된 Lot에서 우선 처리
- **Lot 선택 옵션**: 제조사만 판매 시 특정 Lot 선택 가능 (선택사항)
- **Lot 내 처리**: 선택된 Lot 내에서도 생성 순서대로 FIFO 처리

### 15.2 수량 검증 규칙
- **최소값**: 1 (0 이하 입력 불가)
- **최대값**: 
  - 생산: 100,000 (향후 조정 가능)
  - 출고: 보유 수량까지
- 보유 수량 = 0일 때 출고 시도 → 즉시 에러 표시
- 보유 수량 초과 입력 → 입력 후 검증 에러 표시

### 15.3 이관 오류 회수 규칙
- **권한**: 발송자만 회수 가능
- **기준 시점**: 이관 뭉치 생성(발송) 시점
- **제한 시간**: 24시간
- **사유**: 필수 입력
- **환자 회수**: 환자에게 회수 알림 발송
- **이력 표시**: 회수 이력은 별도 색상으로 명확히 표시 + 사유 함께 표기
- **소유권 복귀**: 이전 소유자(발송자)로 복귀
- **트랜잭션**: 소유권 변경 + 이력 기록이 하나의 트랜잭션

### 15.4 전화번호 정규화
- 형식: 숫자만 저장 (하이픈 제거)
- 예: 010-1234-5678 → 01012345678
- 국가 코드 미포함

### 15.5 환자 소유권
- 시술 완료 시 해당 식별코드의 소유자를 환자 전화번호로 설정
- 동일 전화번호는 동일 환자로 인식, 시술 이력 누적

### 15.6 장바구니 처리
- **영속성**: DB 저장 없음, 해당 페이지 세션 내에서만 일시 유지
- **중복 담기**: 같은 제품을 여러 번 담으면 합산하여 표시
- **수량 수정**: 장바구니에 담은 후 수량 수정 가능

### 15.7 이관 뭉치 처리
- **생성 시점**: 출고 확정 시 자동 생성
- **구성**: 한 번의 출고로 보낸 모든 제품들을 포함
- **UI 표시**: 접기/펼치기 가능
  - 접힌 상태: 발송일, 대상 조직, 제품 종류별 수량, 회수 여부
  - 펼친 상태: Lot별 수량, 제조일자, 사용기한, 개별 식별코드
- **회수 단위**: 이관 뭉치 전체를 회수 (부분 회수 불가)

### 15.8 목록 정렬 기본값
- **재고 목록**: 제품명 순
- **이력 목록**: 최신순
- **이관 뭉치 목록**: 오래된 순

### 15.9 동시성 처리
- 같은 조직의 여러 작업자가 동시에 같은 제품 출고 시도 시:
  1. 먼저 시도된 건이 완료될 때까지 lock 처리
  2. 첫 번째 건 완료 후 수량 확인
  3. 수량 충분 시 다음 건 진행, 부족 시 에러 메시지 출력

### 15.10 세션 관리
- **자동 로그아웃**: 1차에서는 없음
- **중복 로그인**: 1차에서는 제한 없음 (2차에서 단일 세션 강제)
- **관리자 계정**: 개발 과정에서 직접 생성, 페이지를 통한 생성 불가

### 15.11 즉시 이관 원칙
- **Pending 없음**: 모든 출고는 즉시 소유권 이전
- **확인 절차 없음**: 수신자의 수락/확인 액션 불필요
- **오류 대응**: 24시간 내 이관 오류 회수로 처리

## 16. 용어 정의

| 용어 | 정의 |
|------|------|
| **PDO threads** | 미용성형에 사용되는 의료기기 제품 (리프팅 실) |
| **Lot** | 동일 조건에서 생산된 제품 그룹 |
| **UDI-DI** | 의료기기 고유식별코드 - 제품 종류 + 포장 단위별 동일 |
| **UDI-PI** | 의료기기 생산식별코드 - 사용기한 + Lot 번호 결합 |
| **가상 식별코드** | DB에서 관리되는 개별 제품 추적용 고유 번호 |
| **FIFO** | First In First Out - 먼저 생산된 제품이 먼저 출고 |
| **이관 뭉치** | 한 번의 출고로 보낸 제품들을 묶은 단위 |
| **이관 오류 회수** | 발송 후 24시간 이내 발송자가 오발송을 회수하는 기능 |
| **1조직 1계정** | 조직과 계정이 통합되어 조직 내 여러 직원이 하나의 계정 공유 |
